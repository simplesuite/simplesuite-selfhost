# Self-Hosting SimpleBudget

This guide covers how to self-host SimpleBudget on your own infrastructure. You can choose between two deployment modes:

- **Backend-only** — Run your own Supabase-compatible backend while using the hosted frontend on Vercel.
- **Full self-hosting** — Run the entire stack (backend + frontend) on your own machine.

## Prerequisites

- [Docker](https://docs.docker.com/get-docker/) and [Docker Compose](https://docs.docker.com/compose/install/) (v2+)
- `openssl` and `base64` available on your system (for secret generation)
- Git (to clone the repository)

---

## Backend-Only Setup

Use this mode if you want to run your own database and API backend while continuing to use the hosted frontend.

### 1. Clone the repository

```bash
git clone https://github.com/your-org/simplebudget.git
cd simplebudget
```

### 2. Create your environment file

```bash
cp deploy/.env.example deploy/.env
```

### 3. Generate secrets

```bash
cd deploy
./scripts/generate-secrets.sh
cd ..
```

This populates all empty secret values in `deploy/.env` including database passwords, JWT secrets, and API keys. Existing values are preserved.

### 4. Start the backend stack

```bash
docker compose -f deploy/compose.yml up -d
```

This starts Postgres, PostgREST, GoTrue, Kong (API gateway), and runs database migrations automatically.

### 5. Verify the stack is running

```bash
docker compose -f deploy/compose.yml ps
```

All services should show a `healthy` status. The API gateway is available at `http://localhost:8000` by default.

---

## Full Self-Hosting Setup

Use this mode to run both the backend and frontend on your own infrastructure.

### 1. Clone the repository

```bash
git clone https://github.com/your-org/simplebudget.git
cd simplebudget
```

### 2. Create your environment file

```bash
cp deploy/.env.example deploy/.env
```

### 3. Generate secrets

```bash
cd deploy
./scripts/generate-secrets.sh
cd ..
```

### 4. Start the full stack

```bash
docker compose -f deploy/compose.yml -f deploy/compose.frontend.yml up -d
```

This starts the backend services plus a frontend container that builds the React app and serves it via Caddy.

### 5. Access the application

Open `http://localhost:8080` in your browser (or whatever port you configured as `FRONTEND_PORT`).

## Connecting the Hosted Frontend to a Self-Hosted Backend

If you're running backend-only mode, you can point the hosted frontend at your self-hosted backend using localStorage overrides.

### Steps

1. Open the hosted SimpleBudget app in your browser.
2. Open the browser developer tools (F12 or Cmd+Shift+I).
3. Go to the **Console** tab.
4. Set your backend URL and anon key:

```javascript
localStorage.setItem("supabaseUrl", "http://your-server:8000");
localStorage.setItem("supabaseAnonKey", "your-anon-key-from-env-file");
```

5. Reload the page. The frontend will now connect to your self-hosted backend.

### Finding your ANON_KEY

Your `ANON_KEY` is in the `deploy/.env` file after running `generate-secrets.sh`. This key is safe to use in the browser.

### Reverting to the hosted backend

To switch back to the production Supabase instance:

```javascript
localStorage.removeItem("supabaseUrl");
localStorage.removeItem("supabaseAnonKey");
```

Reload the page and the frontend will revert to its default configuration.

## Environment Variables

All configuration is managed through `deploy/.env`. Below is a reference for every variable.

### Public (browser-safe)

| Variable | Default | Description |
|----------|---------|-------------|
| `SUPABASE_PUBLIC_URL` | `http://localhost:8000` | The URL where the Kong API gateway is accessible from the browser. Passed to supabase-js and must be reachable by end users. |
| `ANON_KEY` | *(generated)* | Browser-safe JWT with the "anon" role. Used by supabase-js for unauthenticated API access. Generated by `generate-secrets.sh`. |

### Private (backend-only)

| Variable | Default | Description |
|----------|---------|-------------|
| `SERVICE_ROLE_KEY` | *(generated)* | JWT with the "service_role" role. Has elevated privileges. **Must NEVER be exposed to browser clients.** |
| `JWT_SECRET` | *(generated)* | Cryptographic secret used to sign `ANON_KEY` and `SERVICE_ROLE_KEY` JWTs. **Must NEVER be exposed to browser clients.** |
| `POSTGRES_PASSWORD` | *(generated)* | Password for the Postgres database superuser. **Must NEVER be exposed to browser clients.** |

### Optional

| Variable | Default | Description |
|----------|---------|-------------|
| `DASHBOARD_USERNAME` | `admin` | Username for the Supabase Studio dashboard (if enabled). |
| `DASHBOARD_PASSWORD` | *(generated)* | Password for the Supabase Studio dashboard. Generated by `generate-secrets.sh`. |

### Frontend

| Variable | Default | Description |
|----------|---------|-------------|
| `FRONTEND_PORT` | `8080` | Port on which the self-hosted frontend container is exposed. Only used with `compose.frontend.yml`. |

## Security Notes

Understanding which values are safe to expose is critical for a secure deployment.

### Browser-safe values

- **`SUPABASE_PUBLIC_URL`** — This is just the address of your API gateway. It must be reachable by browsers.
- **`ANON_KEY`** — This is a JWT with the "anon" role. It only grants access that your Row Level Security (RLS) policies allow for unauthenticated users. It is designed to be used in browser clients.

### Values that must NEVER be exposed

- **`SERVICE_ROLE_KEY`** — Bypasses RLS entirely. If leaked, an attacker can read and modify all data in your database.
- **`JWT_SECRET`** — Used to sign JWTs. If leaked, an attacker can forge tokens with any role.
- **`POSTGRES_PASSWORD`** — Direct database access. If leaked, an attacker has full control of your data.

### Best practices

- Never commit `deploy/.env` to version control (it's in `.gitignore`).
- The frontend container only receives `SUPABASE_PUBLIC_URL` and `ANON_KEY` — private secrets are never passed to it.
- If you expose your backend to the internet, use Tailscale serve or a reverse proxy with HTTPS (see [Remote Access](#remote-access)).

## Updating and Redeploying

When a new version of SimpleBudget is released:

### 1. Pull the latest changes

```bash
git pull origin main
```

### 2. Rebuild containers

For backend-only:

```bash
docker compose -f deploy/compose.yml up -d --build
```

For full self-hosting:

```bash
docker compose -f deploy/compose.yml -f deploy/compose.frontend.yml up -d --build
```

The `--build` flag ensures the frontend container is rebuilt with the latest source code. Database migrations run automatically on startup, so your schema will be updated.

### 3. Verify the update

```bash
docker compose -f deploy/compose.yml ps
```

Check that all services are healthy. If a migration fails, see the [Troubleshooting](#troubleshooting) section.

## Remote Access

### Recommended: Tailscale serve

[Tailscale](https://tailscale.com/) provides the simplest way to access your self-hosted instance from other devices without configuring HTTPS certificates or reverse proxies.

```bash
# Expose the API gateway to your Tailscale network
tailscale serve 8000

# Expose the frontend (if full self-hosting)
tailscale serve 8080
```

This gives you a `https://your-machine.tailnet-name.ts.net` URL that is accessible from any device on your Tailscale network with automatic HTTPS.

**Why Tailscale?**

- No port forwarding or firewall configuration needed.
- Automatic HTTPS with valid certificates.
- Access is restricted to your Tailscale network by default.
- No need to configure a reverse proxy like nginx or Traefik.

After setting up Tailscale serve, update `SUPABASE_PUBLIC_URL` in your `deploy/.env` to the Tailscale HTTPS URL and restart the stack.

## Troubleshooting

### Postgres fails to start or stays unhealthy

**Symptoms:** Other services don't start. `docker compose ps` shows postgres as `unhealthy`.

**Solutions:**

1. Check Postgres logs:
   ```bash
   docker compose -f deploy/compose.yml logs postgres
   ```
2. Ensure `POSTGRES_PASSWORD` is set in `deploy/.env`. Run `./deploy/scripts/generate-secrets.sh` if needed.
3. If the data volume is corrupted, remove it and restart (this deletes all data):
   ```bash
   docker compose -f deploy/compose.yml down -v
   docker compose -f deploy/compose.yml up -d
   ```

### Migration errors

**Symptoms:** The `migrations` service exits with a non-zero code. PostgREST may not start.

**Solutions:**

1. Check migration logs:
   ```bash
   docker compose -f deploy/compose.yml logs migrations
   ```
   The error output includes the failing filename and SQL error details.
2. Fix the failing SQL file in `supabase/migrations/` and restart:
   ```bash
   docker compose -f deploy/compose.yml up -d
   ```
3. If you need to re-run migrations from scratch, drop the tracking table:
   ```bash
   docker compose -f deploy/compose.yml exec postgres psql -U postgres -c "DROP TABLE IF EXISTS schema_migrations;"
   docker compose -f deploy/compose.yml restart migrations
   ```

### Frontend not loading

**Symptoms:** Browser shows a connection error or blank page on port 8080.

**Solutions:**

1. Verify the frontend container is running:
   ```bash
   docker compose -f deploy/compose.yml -f deploy/compose.frontend.yml ps frontend
   ```
2. Check frontend logs:
   ```bash
   docker compose -f deploy/compose.yml -f deploy/compose.frontend.yml logs frontend
   ```
3. Ensure `FRONTEND_PORT` is not in use by another application.
4. Verify Kong is healthy (the frontend depends on it):
   ```bash
   docker compose -f deploy/compose.yml ps kong
   ```

### config.json not found (404 in browser console)

**Symptoms:** Browser console shows `[config] config.json not found`. The app may fall back to build-time variables or show a connection error.

**Solutions:**

1. This is **expected behavior** on the hosted Vercel frontend — it falls back to build-time env vars.
2. For self-hosted frontend, check that `SUPABASE_PUBLIC_URL` and `ANON_KEY` are set in `deploy/.env`:
   ```bash
   grep -E "^(SUPABASE_PUBLIC_URL|ANON_KEY)=" deploy/.env
   ```
3. Restart the frontend container to regenerate config.json:
   ```bash
   docker compose -f deploy/compose.yml -f deploy/compose.frontend.yml restart frontend
   ```
4. Verify config.json is being served:
   ```bash
   curl http://localhost:8080/config.json
   ```

### Kong returns 502 or connection refused

**Symptoms:** API requests fail with 502 Bad Gateway.

**Solutions:**

1. Check that PostgREST and GoTrue are healthy:
   ```bash
   docker compose -f deploy/compose.yml ps postgrest gotrue
   ```
2. Check Kong logs:
   ```bash
   docker compose -f deploy/compose.yml logs kong
   ```
3. Restart the stack:
   ```bash
   docker compose -f deploy/compose.yml down
   docker compose -f deploy/compose.yml up -d
   ```

## Architecture Overview

```
┌─────────────────────────────────────────────────────┐
│                  Docker Compose                       │
│                                                      │
│  ┌──────────┐    ┌───────────┐    ┌──────────────┐  │
│  │ Postgres │◄───│ PostgREST │    │   GoTrue     │  │
│  │  :5432   │    │   :3000   │    │   :9999      │  │
│  └────┬─────┘    └─────┬─────┘    └──────┬───────┘  │
│       │                 │                 │          │
│       │          ┌──────┴─────────────────┘          │
│       │          │                                   │
│       │    ┌─────┴─────┐                             │
│       │    │   Kong    │ ◄── Browser / supabase-js   │
│       │    │   :8000   │                             │
│       │    └───────────┘                             │
│       │                                              │
│  ┌────┴──────────┐    ┌───────────────┐             │
│  │  Migrations   │    │   Frontend    │ (optional)  │
│  │  (runs once)  │    │ Caddy :8080   │             │
│  └───────────────┘    └───────────────┘             │
└─────────────────────────────────────────────────────┘
```

- **Kong** is the single entry point for all API requests. It routes `/rest/v1/*` to PostgREST and `/auth/v1/*` to GoTrue.
- **Migrations** run once at startup and apply any new SQL files from `supabase/migrations/`.
- **Frontend** (optional) builds the React app and serves it with runtime configuration via `config.json`.
